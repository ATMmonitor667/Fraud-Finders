WHAT THIS DATASET IS ABOUT
--------------------------

This dataset simulates 90 days of banking activity at Crestline Community Bank.
It contains 5,000 transactions across approximately 294 customer accounts, mixed
with everyday merchant purchases and a hidden coordinated fraud ring.

The core challenge: a fraud ring deliberately kept every transfer below typical
alert thresholds. No single transaction looks alarming on its own. The suspicious
pattern only becomes visible when you connect account-to-account flows, timing,
amount bands, and shared device fingerprints across the graph.

This is the competition dataset for Track 02 (Fraud Watch). Judges verify
findings against a hidden answer key. The README hints at one ring of roughly
12 accounts with total exposure of about $161,751.

WHAT WAS CLEANED
----------------

The raw data was already high quality. The cleaning pipeline:

  1. Normalised column names (txn_id -> transaction_id)
  2. Parsed timestamps and account_open_date as datetimes
  3. Coerced amounts to numeric (stripped currency symbols if present)
  4. Trimmed whitespace on string fields
  5. Removed invalid rows: nulls, zero/negative amounts, self-transfers,
     duplicate transaction IDs (0 rows removed — none were bad)
  6. Added derived features for fraud analysis

Result: 5,000 rows retained (100%), expanded to 21 columns in the clean file.

THE HIDDEN FRAUD RING
---------------------

The fraud activity is concentrated in account-to-account transfers, not merchant
payments. Key characteristics discovered in the data:

  - 250 inter-account transfers, all between $402 and $899
  - All 250 transfers occur between 2:00 AM and 4:00 AM
  - 9 accounts form the visible transfer network:
      AC-0001, AC-0002, AC-0003, AC-0005, AC-0006, AC-0007,
      AC-0009, AC-0010, AC-0011
  - Total ring exposure: $161,750.90 (matches the README hint of ~$161,751)
  - Five accounts opened in a tight window (Feb 10–18, 2026):
      AC-0010, AC-0001, AC-0011, AC-0009, AC-0005
  - Transfers follow near-regular intervals (~42 transactions per edge pair)
  - Two device fingerprints are shared across unrelated account pairs
      (potential mule or linked accounts):
      DV-52001 shared by AC-0131 and AC-0192
      DV-31829 shared by AC-0127 and AC-0045

  Transfer graph (directed edges):
    AC-0001  -> AC-0002   (42 txns)
    AC-0005  -> AC-0006   (42 txns)
    AC-0005  -> AC-0009   (41 txns)
    AC-0009  -> AC-0007   (42 txns)
    AC-0010  -> AC-0011   (42 txns)
    AC-0011  -> AC-0003   (41 txns)

  AC-0002, AC-0003, AC-0006, and AC-0007 appear only as receivers — they never
  initiate transfers in this dataset but receive funds from the ring.


DISTRACTORS AND FALSE-POSITIVE TRAPS
------------------------------------

Most transactions (95%) are normal merchant spending in the $10–$100 range.
The dataset also includes legitimate-looking anomalies designed to fool simple
rules:

  - Large single purchases up to ~$15,000 (not fraud)
  - Occasional late-night merchant transactions (not fraud alone)
  - High-frequency everyday spenders (not fraud alone)
  - Foreign IP regions on a handful of transactions (21 total)
  - Shared devices on non-ring accounts (may or may not be suspicious)

A naive threshold rule (e.g. "flag anything over $500" or "flag all 2–4 AM
activity") will produce many false positives. Effective detection requires
combining graph structure, timing, amount bands, and device linkage.


PROBLEMS THAT CAN BE SOLVED WITH THIS DATA
------------------------------------------

  1. Fraud ring detection
     Build a directed graph from account-to-account transfers and find cyclic
     money flows that stay under alert thresholds.

  2. Coordinated account identification
     Cluster accounts opened in the same time window with similar transaction
     patterns (late-night, $400–$900 band, regular intervals).

  3. Device fingerprint analysis
     Flag accounts sharing the same device_id as potential linked identities.

  4. Anomaly scoring
     Score accounts by combining late_night_pct, suspect_range_pct, transfer
     frequency, and graph centrality rather than single-feature thresholds.

  5. Exposure quantification
     Sum total funds moved through the ring ($161,751) for case briefs and
     regulatory reporting.

  6. Analyst workflow automation
     Ingest CSV -> build knowledge graph -> detect rings -> score risk ->
     generate a human-readable case brief (the RingAlert project goal).
